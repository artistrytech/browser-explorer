# 追加改修 設計書 — Git / Explorer 機能拡張

## 0. ドキュメントについて

本書は、Explorer ベースのファイル管理 + テキスト編集 + Git アプリ（ローカルサーバー前提、
システム git のラッパー、URL 駆動ナビゲーション、サーバー側 SQLite 永続化、WebSocket による
変更通知）に対する追加改修をまとめた設計書である。以下 7 件を扱う。

| # | 改修 | 種別 |
| --- | --- | --- |
| 1 | Git 管理下のファイル/フォルダで「ログを表示」= **パス絞り込みログ** | Git |
| 2 | マージ競合中に「競合を解消」= **競合ファイル一覧 → 3-way 解消ツール**（TortoiseGitMerge 相当） | Git |
| 3 | 非リポジトリのフォルダで「Git Clone」= **クローンダイアログ**（TortoiseGit 相当 UI） | Git |
| 4 | フォルダ/空白の右クリックに **OS 連携**（Finder/Explorer で開く・ターミナル/コマンドプロンプトで開く） | OS |
| 5 | **Git グラフ表示**（コミット DAG のレーン描画）を新規設計 | Git |
| 6 | Explorer の**フォーカス/選択位置を sessionStorage で復元**（ブラウザバック対応） | Explorer |
| 7 | フォルダの**クイックアクセスへのピン止め / 解除**（解除は確認あり） | Explorer |

## 1. パス指定 Git ログ「ログを表示」

### 1.1 概要

Git 管理下のファイル/フォルダを右クリックすると「ログを表示」を表示し、
ログ/履歴ビューアを**対象パスでフィルタした状態**で開く。そのパスに触れたコミットのみを表示する。

- ファイル対象: そのファイルの変更履歴（リネーム追跡付き）。
- フォルダ対象: フォルダ配下のいずれかに触れたコミット。

### 1.2 コンテキストメニュー

- 表示条件: 対象が Git 管理下（`is-repo` = true）のとき表示。ファイル/フォルダ双方で表示。

### 1.3 API

`GET /api/git/log` に `path`（リポジトリルート相対）・`follow` パラメータを持たせる。

| パラメータ | 説明 |
| --- | --- |
| `repo` | リポジトリルート |
| `path` | 絞り込み対象の相対パス（省略時は全体） |
| `follow` | ファイル時 true でリネーム追跡（`--follow`） |
| `limit` / `skip` | 件数 / ページング |

実行する git（概念）:

```
# ファイル: リネーム追跡
git log --follow -- <relpath>
# フォルダ: 配下に触れたコミット
git log -- <relpath>/
```

- パスは必ずリポジトリルート相対に正規化してから渡す（ルート外は拒否）。
- `simple-git` では `git.log({ file })` か `git.raw(['log', ...,'--', relpath])`。

### 1.4 UI と注意点

- ビューアヘッダに絞り込み中パス（例: `src/index.ts の履歴`）と「絞り込み解除」トグルを表示。
- `--follow` は**単一ファイルのみ**有効（フォルダには付けない）。
- パス絞り込み時は履歴が疎になるため、**グラフではなく線形リスト表示**を既定とする
  （グラフ描画は全体表示時のみ。§5 参照）。

## 2. マージ競合の解消「競合を解消」

### 2.1 概要

リポジトリが**マージ/リベース/cherry-pick 進行中**かつ競合がある場合、フォルダの右クリックに
「競合を解消」を表示する。クリックで**競合ファイル一覧**を開き、ファイル選択で
**3-way マージ解消ツール（TortoiseGitMerge 相当 UI）**を開く。

### 2.2 競合状態の検出

| 判定 | 方法 |
| --- | --- |
| 進行中か | `.git/MERGE_HEAD` / `REBASE_HEAD` / `CHERRY_PICK_HEAD` の有無、または `git status` |
| 競合ファイル | `git diff --name-only --diff-filter=U`、または `git status --porcelain=v2` の unmerged 行 |
| 競合種別 | ステージ 1/2/3 の有無で判定（both modified / add-add / delete-modify 等） |

- WebSocket の Git 状態通知に `{ inProgress, conflictedPaths[] }` を含め、
  コンテキストメニューが同期的に「競合を解消」の要否を判断できるようにする。
- 競合ファイルは Explorer 上で **⚠ オーバーレイ**が付く。

### 2.3 競合ファイル一覧（パネル）

```
┌──────── 競合を解消: src/ (3件) ────────────────────────┐
│  ⚠ src/app.ts        both modified                     │
│  ⚠ src/util/date.ts  both modified                     │
│  ⚠ src/logo.png      both added (binary)               │
│ ─────────────────────────────────────────────────────  │
│  [ すべて自分(ours)を採用 ] [ すべて相手(theirs)を採用 ] │
│  [ マージを中止 ]                        残り: 3 件      │
└────────────────────────────────────────────────────────┘
```

- 各行: パス・競合種別・（バイナリ表示）。クリックで解消ツールを開く。
- 解決済み行はチェック表示、残件数を更新。一括操作と「マージを中止」を用意。

### 2.4 3-way マージ解消ツール（TortoiseGitMerge 相当 UI）

```
┌──────────────── src/app.ts の競合を解消 ────────────────┐
│  自分 (Mine / HEAD)        │      相手 (Theirs / MERGE_HEAD)│  ← 上段：ours / theirs
│  <差分ハイライト>           │      <差分ハイライト>           │
│  [◀ この塊を採用] [両方採用] │  [この塊を採用 ▶]              │  ← 競合ブロック毎の採用
├────────────────────────────┴────────────────────────────┤
│  統合結果 (編集可能 / Monaco)                              │  ← 下段：結果ペイン（手編集可）
├──────────────────────────────────────────────────────────┤
│ 未解決 2/5  [自分を全採用][相手を全採用][解決としてマーク] │
└──────────────────────────────────────────────────────────┘
```

- 上段に「自分(ours)」「相手(theirs)」を左右対比（任意で「ベース(base)」3 ペインも可）、
  下段に**編集可能な統合結果ペイン**。TortoiseGitMerge 準拠の構成。
- 各競合ブロックに「自分を採用 / 相手を採用 / 両方採用（順序選択）」ボタン。結果ペインへ反映。
- 結果ペインは Monaco で手編集可。次/前の未解決競合へジャンプ、未解決件数を常時表示。
- ブラウザ上で TortoiseGitMerge 本体は動かないため、Monaco の diff 表示（base↔ours / base↔theirs）
  ＋独自の結果ペインで**同等 UI をエミュレート**する。

### 2.5 バージョン取得

`git show :1:path`（base/祖先）、`:2:path`（ours/HEAD）、`:3:path`（theirs/MERGE_HEAD）と作業ツリー
（競合マーカー入り）を取得。ステージ欠落やバイナリ判定を含めて返す。

### 2.6 解決フロー

1. フォルダ右クリック →「競合を解消」→ 一覧 → ファイル選択 → 3-way ツール。
2. ブロック採用/手編集で結果を作成 →「解決としてマーク」（結果保存 + `git add`）。
3. 全件解決 →「マージを完了（コミット）」。いつでも「マージを中止」で開始前へ戻せる。

### 2.7 エッジケース

- **バイナリ競合**: 3-way 不可 →「自分を採用 / 相手を採用」のみ。
- **delete/modify**: 「残す（変更側採用）/ 削除」の選択。
- **add/add**: base 無し → ours/theirs 2 ペイン対比。
- 進行状態が解消されたら（WS で `inProgress=false`）ツール/メニューを自動的に閉じる。

## 3. Git Clone（非リポジトリ時）

### 3.1 概要

**Git 管理外のフォルダ/空白**を右クリックすると「Git Clone…」を表示し、クローンダイアログを開く。
TortoiseGit のクローンダイアログに倣った UI とする。

### 3.2 コンテキストメニュー

- 表示条件: 対象パスが**どのリポジトリにも属さない**（`is-repo` = false）とき表示。

### 3.3 ダイアログ UI（TortoiseGit 相当）

```
┌──────────────── Git Clone ─────────────────────────────┐
│ URL:        [ https://github.com/user/repo.git       ] │
│ 保存先:      [ /Users/me/work/repo            ] [参照] │
│ ─ オプション ─────────────────────────────────────────  │
│ [ ] ブランチ指定   [ main            ]                  │
│ [ ] Shallow (depth) [ 1  ]                              │
│ [x] サブモジュールも取得 (--recursive)                  │
│ [x] クローン後に開く                                    │
│                              [ キャンセル ] [ Clone ]  │
└────────────────────────────────────────────────────────┘
```

| フィールド | 説明 |
| --- | --- |
| URL | クローン元（https / ssh） |
| 保存先 | 既定は右クリックしたフォルダ配下に `repo 名` を補完 |
| ブランチ指定 | `--branch` で特定ブランチ/タグ |
| Shallow (depth) | `--depth N` で浅いクローン |
| サブモジュール | `--recursive` |
| クローン後に開く | 完了後にそのフォルダへ遷移 |

### 3.4 API と進捗

- `POST /api/git/clone`：`{ url, dir, branch?, depth?, recursive?, bare? }`。
- クローンは長時間になり得るため、**進捗を WebSocket でストリーム**（`git clone --progress` の出力を
  中継）。ダイアログは進捗バー/ログを表示、完了/失敗を通知。
- 認証はシステム git に委譲（credential helper / SSH agent）。非対話実行のため資格情報が未構成だと
  失敗する。その場合は「ターミナルで一度認証を通してください」と案内する（アプリは資格情報を持たない）。
- 成功時: `recursive` 等を反映しつつリポジトリ一覧（DB）へ登録。「開く」指定なら遷移。

## 4. OS 連携メニュー（ファイル以外＝フォルダ/空白）

### 4.1 概要

**対象がファイルでない場合**（フォルダ、または一覧の空白＝カレントフォルダ）の右クリックに、
ホスト OS に応じた「OS のファイルマネージャで開く」「OS のターミナルで開く」を追加する。

| OS | メニュー項目 |
| --- | --- |
| macOS | 「Finder で開く」「ターミナルで開く」 |
| Windows | 「Explorer で開く」「コマンドプロンプトで開く」 |

※ ここで言う「ターミナルで開く」は**OS 標準ターミナルを外部起動する**もので、
アプリ内蔵ターミナルではない（内蔵ターミナルは非対応）。

### 4.2 OS 判定とラベル出し分け

- サーバーの `process.platform` を `GET /api/os/platform` で返し、クライアントがラベルを出し分ける。
- 主対象は macOS だが、Windows で起動された場合もメニューが適切に切り替わる。

### 4.3 API

| メソッド | エンドポイント | 説明 |
| --- | --- | --- |
| GET | `/api/os/platform` | `darwin` / `win32` 等を返す |
| POST | `/api/os/open-in-file-manager` | `{ path }` を OS のファイルマネージャで開く |
| POST | `/api/os/open-in-terminal` | `{ path }` を OS のターミナルで開く |

起動コマンド（概念）:

| OS | ファイルマネージャ | ターミナル |
| --- | --- | --- |
| macOS | `open <path>` | `open -a Terminal <path>` |
| Windows | `explorer <path>` | `cmd` を対象パスをカレントにして起動 |

### 4.4 セキュリティ

- 外部プロセス起動は**引数配列で spawn**し、シェル文字列補間を避ける（インジェクション防止）。
- パスは実在ディレクトリに限定し、正規化してから渡す。

## 5. Git グラフ表示（新規設計）

現状グラフ表示は未実装のため、コミット DAG（有向非巡回グラフ）をレーン（レール）描画するための
設計を新規に定義する。GitKraken / VS Code の Git Graph 相当の見た目を目標とする。

### 5.1 表示イメージ

```
●─┐        a1b2  main, HEAD   feat: add parser        Alice  2026/..
│ ●        c3d4  origin/main  fix: null check         Bob    2026/..
● │        e5f6               merge branch 'topic'    Alice  2026/..
├─●        7890               topic: refactor         Carol  2026/..
● │        ...
```

左に**グラフ（レーン + ノード + エッジ）**、右に**コミット行（ハッシュ・ref・メッセージ・作者・日時）**を
並べる。行高は固定。ref（ブランチ/タグ/HEAD）はチップとして表示。

### 5.2 バックエンド（グラフ用ログ取得）

`GET /api/git/graph?repo=&all=&limit=&skip=`

- 実行する git（概念）。区切りに制御文字を使い安全にパース:
  ```
  git log [--all] --topo-order \
    --pretty=format:%H%x1f%P%x1f%an%x1f%aI%x1f%D%x1f%s%x1e --max-count=N --skip=M
  ```
  - `%H` ハッシュ / `%P` 親ハッシュ（空白区切り、複数＝マージ）/ `%an` 作者 / `%aI` ISO 日時 /
    `%D` ref 名（HEAD・ブランチ・タグ）/ `%s` 件名。
- `all=true` で全ブランチ、false で現在ブランチのみ。
- 返却は**コミット配列**（各要素に `hash, parents[], author, date, refs[], subject`）。
  レーン割り当てはクライアント側で行う（描画都合のため）。

### 5.3 レーン割り当てアルゴリズム（クライアント）

表示順（トポロジカル、上が新しい）に上から処理し、各コミットの**列（レーン）**を決める。

状態: `lanes` = 各レーンが「次に描く予定のコミットハッシュ」を保持する配列。

各コミット `C` について:

1. **列の決定**: `C.hash` を予約しているレーンのうち最左を `C` の列とする。
   無ければ空きレーンを割り当てる（＝ブランチ先端 / HEAD）。
2. **合流**: `C.hash` を予約している他のレーンは `C` の列へ合流（＝マージ流入エッジ）。合流元は解放。
3. **親の展開**:
   - **第 1 親**は `C` の列を引き継ぐ（そのレーンの予約を第 1 親のハッシュへ）。
   - **追加の親**（マージ元）は、その親を予約済みのレーンがあれば再利用、無ければ右側に新規レーン割当。
     → `C` の列から各親レーンへの**分岐エッジ**を描く。
4. **通過レーン**: `C` と無関係で他ハッシュを予約中のレーンは、その行では**縦の直線**を描くだけ。

- **色**: レーンごとにパレットを循環割当し、ブランチを色で識別できるようにする。
- この方式は `git log --graph` 系の一般的なレーン圧縮と同等。

### 5.4 レンダリング

- 左のグラフは**SVG**で描画。ノード＝円、エッジ＝直線/ベジェで隣接行間を接続。
- 右のコミットリストと**同一行高で厳密に整列**させる。
- **仮想スクロール**：可視行＋バッファのみ描画。ただしレーン状態は上から連続計算が必要なため、
  読み込み済みコミットのレーンを**先頭から算出してキャッシュ**し、可視範囲だけ SVG 化する。
- **ページング**：`skip` で追記ロード。レーン状態の連続性のため、読み込み済み全体を対象に
  レーンを再計算（or 安定点からの増分計算）する。

### 5.5 装飾・操作

- ref チップ: `HEAD`・ローカル/リモートブランチ・タグを色分けチップで表示。
- 操作（既存の履歴ビューアと共通）: コミット選択で差分表示、右クリックで checkout / reset /
  cherry-pick / タグ付け等。ホバーで当該コミットの経路をハイライト（任意）。

### 5.6 パス絞り込みとの関係

§1 のパス絞り込みログでは履歴が疎になりグラフが意味を持たないため、
**グラフは全体表示（絞り込み無し）時のみ**描画し、絞り込み時は線形リストにフォールバックする。

## 6. フォーカス/選択位置の復元（sessionStorage）

### 6.1 概要

Explorer でフォーカス/選択していた項目の位置を **sessionStorage** に保持し、
**ブラウザバック/フォワードやリロードで同じフォルダに戻った際に復元**する。

ナビゲーションは URL 駆動（カレントパスは URL、戻る/進む=ブラウザ履歴）。本機能はその上で
「戻った先でどの項目にフォーカスが当たっていたか」を再現する。

> sessionStorage はタブ内で保持され、**リロード・戻る/進むでは維持、タブを閉じると破棄**される。
> 「ブラウザバックで復元・別タブや再訪では初期化」という要件に合致する。

### 6.2 保存データ

```
key:   exp:focus:<正規化パス>
value: { focused:"childName", selected:["a","b"], scrollTop:1234, ts:<epoch> }
```

### 6.3 保存タイミング

- 選択/フォーカス変更時（デバウンス書き込み）。
- **サブフォルダへ入る直前**に「今アクティブな子項目」を親パスの focus として保存
  → 戻った時、さっき入った子フォルダがハイライトされる（Windows Explorer 挙動）。
- 離脱/リロード直前（`visibilitychange` / `pagehide`）にも最新化。

### 6.4 復元タイミング（URL・履歴と連動）

- 一覧の読み込み完了後（クリック遷移・アドレスバー入力・**popstate（戻る/進む）**いずれでも）、
  現在パスのキーを読み、フォーカス・選択・スクロール位置を復元。
- キーが無い初回訪問は先頭にフォーカス（デフォルト）。

### 6.5 エッジケース

- **項目消失/リネーム**（WS 変更通知）: 保存名が無ければ近い index にフォールバック、無ければ解除。
- **スクロール復元**: 仮想スクロールのため、行の実描画後に `scrollTop` を適用。
- **キー肥大化**: タブ閉鎖で破棄されるが、上限（例: 直近 200 パス）を設け `ts` の古い順に間引く。
- **多重タブ**: sessionStorage はタブ独立のため干渉しない。

## 7. クイックアクセスのピン止め / 解除

### 7.1 概要

フォルダをサイドバーの**クイックアクセス（ブックマーク）にピン止め**、および**解除**する。
クイックアクセスはサーバー側 DB（`favorites` テーブル）に永続化される既存の仕組みを用いる。
「よく使う場所」の登録手段はこのピン止めのみ（ワークスペース概念は持たない）。

### 7.2 ピン止め（コンテキストメニュー）

- 表示条件: 対象が**フォルダ**のとき表示（ファイルは対象外）。
  - 未ピンのフォルダ →「クイックアクセスにピン止め」。
  - 既にピン済みのフォルダ →「ピン止めを解除」に切り替え（§7.3 と同じ確認フローへ）。
- 動作: フォルダのパスと**ラベル（既定＝フォルダ名）**を追加。
  - パス重複は登録しない（既にあれば何もせずトーストで通知）。
  - 追加位置は末尾（`sort` を採番）。並べ替えはドラッグで可能。
- 反映: サイドバーのクイックアクセスへ即時追加。

### 7.3 解除（クイックアクセス上・確認あり）

- 起点: クイックアクセス項目の右クリック「ピン止めを解除」、またはホバー時に現れる ✕ ボタン。
  （フォルダ本体の右クリックで既にピン済みの場合も同じ解除フロー。）
- **解除の前に確認ダイアログを必須**とする。

```
┌──────────── ピン止めの解除 ────────────┐
│ 「project」をクイックアクセスから外し   │
│ ますか？                                │
│                                        │
│ ※ ブックマークを外すだけで、フォルダ    │
│   本体は削除されません。                │
│                     [ キャンセル ] [外す]│
└────────────────────────────────────────┘
```

- 確認後に削除し、サイドバーから即時除去。
- **重要**: 解除はブックマークの除去のみで、**実フォルダには一切影響しない**旨をダイアログに明記
  （誤ってフォルダ削除と混同させない）。

### 7.4 API

| メソッド | エンドポイント | 用途 |
| --- | --- | --- |
| GET | `/api/quickaccess` | 一覧取得（サイドバー描画用） |
| POST | `/api/quickaccess` | ピン止め追加（`{ path, label? }`、パス重複は無視） |
| DELETE | `/api/quickaccess` | 解除（`{ path }`） |
| POST | `/api/quickaccess/reorder` | 並べ替え（任意） |

- DB: `favorites(id, path UNIQUE, label, sort)`。追加時は末尾 `sort` を採番。
- 単一ユーザーのため排他制御は不要。更新後はサイドバーを再取得（or 楽観的更新）。

### 7.5 エッジケース

- **重複ピン**: 既存パスの再ピンは無視し「すでにピン止めされています」と通知。
- **ピン先が移動/削除された**（WS 変更通知）: 項目は残しつつ「見つかりません」表示にし、
  クリック時に案内、または解除を促す。
- **無効パス**: 存在しない/ルート外のパスのピンは拒否。
- **ラベル**: 既定はフォルダ名。将来的にリネーム（表示名変更）を許容する余地を残す。

## 8. コンテキストメニュー構成（文脈別の出し分け）

対象と状態に応じて項目を出し分ける。主要な文脈依存項目は以下。

| 項目 | 表示条件 |
| --- | --- |
| エディタで開く | 対象がファイル |
| Finder/Explorer で開く | 対象が**ファイル以外**（フォルダ/空白） |
| ターミナル/コマンドプロンプトで開く | 対象が**ファイル以外** |
| クイックアクセスにピン止め | 対象が**フォルダ**かつ未ピン |
| ピン止めを解除（確認あり） | 対象が**フォルダ**かつピン済み／クイックアクセス項目上 |
| ログを表示 | 対象が**Git 管理下**（ファイル/フォルダ） |
| コミット / 差分 / プッシュ 等（既存 Git 群） | 対象が Git 管理下 |
| 競合を解消 | リポジトリが**進行中 かつ 配下に競合あり** |
| Git Clone… | 対象が**Git 管理外**のフォルダ/空白 |
| コピー / 切り取り / 貼り付け / 削除 / 名前変更 / プロパティ | 常時（対象種別に応じ） |
